# Kubernetes Namespaces

## What is a Namespace?

A Namespace provides an extra layer of isolation **within a single cluster**, letting you logically separate objects and resources for different teams, environments, or applications — without needing separate clusters.

**Interview one-liner:** "Namespaces are a way to divide cluster resources between multiple users or teams — they give logical isolation, not physical isolation."

- If you don't specify a namespace, Kubernetes places the object in the **`default`** namespace automatically.
- When a cluster is provisioned, Kubernetes automatically creates a few built-in namespaces.
- You can attach different **RBAC permissions and quotas** to each namespace, so teams only get access to what they need.

---

## Default Namespaces

| Namespace | Purpose |
|---|---|
| `default` | Where resources land if you don't specify a namespace |
| `kube-system` | Reserved for Kubernetes **control plane components** (kube-dns/coredns, kube-proxy, etc.) — by default, control plane components are created here |
| `kube-public` | Readable by all users (including unauthenticated); mostly used for cluster info that should be publicly accessible |
| `kube-node-lease` | Holds **node lease objects** — used by the control plane to check node heartbeat/availability |

```bash
kubectl get ns          # list all namespaces
kubectl get all         # only shows resources in the CURRENT namespace (default), NOT all namespaces
```

**Correction from your notes:** `kubectl get all` does **not** show everything across all namespaces — it only shows resources in whichever namespace your context is pointed at (default, unless changed). To see everything across all namespaces, you need `-A` / `--all-namespaces`.

```bash
kubectl get all -A
kubectl get all --all-namespaces
kubectl get all -n kube-system          # resources inside a specific namespace
kubectl get all --namespace=kube-system # same, long form
```

**kube-dns / CoreDNS service** — runs inside `kube-system` and is responsible for resolving IPs to hostnames (service discovery) within the cluster.

## Namespace Scope — What CAN and CANNOT Be Namespaced

This is a commonly asked interview point that's easy to miss:

- **Namespaced resources**: Pods, Deployments, Services, ConfigMaps, Secrets, ReplicaSets, PVCs, etc.
- **Cluster-scoped resources (NOT namespaced)**: Nodes, PersistentVolumes, StorageClasses, ClusterRoles, ClusterRoleBindings, Namespaces themselves.

```bash
kubectl api-resources --namespaced=true    # list namespaced resource types
kubectl api-resources --namespaced=false   # list cluster-scoped resource types
```

## Creating a Namespace

### Imperative way
```bash
kubectl create ns demo
```

### Declarative way (YAML)
```yaml
# ns.yml
apiVersion: v1
kind: Namespace
metadata:
  name: demo
```

```bash
kubectl apply -f ns.yml
kubectl get ns
kubectl delete ns demo
```

## Deploying Into a Namespace

```bash
kubectl create deploy nginx-demo --image=nginx -n demo
```

```bash
kubectl get deploy -n demo
kubectl get pods -n demo
```

> If you don't specify `-n <namespace>`, the resource is created in `default` automatically — and you won't need `-n` to view it either, since `default` is your context's default namespace.

## Pod-to-Pod Communication Within a Namespace

Two pods in the **same namespace** can talk to each other directly using just the **hostname** (pod name / service name) — no need for the full domain.

```bash
kubectl get pods -n demo
kubectl exec -it <pod-name> -n demo -- sh
  # inside the pod:
  ping <other-pod-hostname>     # works if ping is installed
  curl <ip-address>             # always works, IP is reachable
```

> **Important clarification (missing from the original notes):** Namespaces do **not** block network traffic by default. Pods across *different* namespaces **can still reach each other by IP** — Kubernetes' flat pod network allows this unless a **NetworkPolicy** is explicitly applied to restrict it. What breaks across namespaces is **hostname-only resolution** — that's a DNS naming scope issue, not a network isolation feature. This distinction ("isolation vs. networking") comes up a lot in interviews.

## Exposing a Deployment via a Service

A Service gives pods a **stable identity** to be reached by, since pod IPs are ephemeral (they change every time a pod restarts/reschedules).

### Imperative way
```bash
kubectl expose deploy nginx-demo --name=svc-demo --port=80 -n demo
kubectl get svc -n demo
```

## DNS Resolution & FQDN (Fully Qualified Domain Name)

The file responsible for IP-to-DNS resolution inside the cluster/host:
```bash
cat /etc/resolv.conf
```

**FQDN format:**
```
<service-name>.<namespace>.svc.cluster.local
```

Example:
```bash
curl svc-test.default.svc.cluster.local
```

| Accessing from... | You can use |
|---|---|
| Same namespace | Just the service/pod name (short hostname) |
| Different namespace | Full FQDN — `<service>.<namespace>.svc.cluster.local` |
| Anywhere in the cluster | Direct pod/service **IP address** always works |

## Key Takeaways (Interview Summary)

1. **Namespaces = logical isolation**, not network isolation. They organize resources and let you scope RBAC and ResourceQuotas — they don't block traffic by default.
2. Four built-in namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.
3. Not everything is namespaced — nodes, PVs, and StorageClasses are **cluster-scoped**.
4. `kubectl get all` only shows the *current* namespace; use `-A` for everything.
5. Pods in the same namespace → reach each other by short hostname.
   Pods across namespaces → need the FQDN (`svc.namespace.svc.cluster.local`), but **IP always works** regardless of namespace.
6. Services exist because **pod IPs are not static** — they give a stable endpoint to reach a set of pods.
7. CoreDNS (running in `kube-system`) is what performs this internal IP-to-hostname resolution.
