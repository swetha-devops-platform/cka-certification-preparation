# Kubernetes Autoscaling 

# Why Autoscaling Matters

- Kubernetes autoscaling automatically adjusts compute resources based on demand, so apps stay performant without wasting cost. 

- There are **3 layers** of autoscaling, and each solves a different problem:

| Type | Scales | Based On | Layer |
|---|---|---|---|
| **HPA** (Horizontal Pod Autoscaler) | Number of pod replicas | CPU/Memory/Custom metrics | Pod |
| **VPA** (Vertical Pod Autoscaler) | CPU/Memory requests-limits of a pod | Historical usage | Pod |
| **Cluster Autoscaler (CA)** | Number of nodes | Pending pods (unschedulable) | Node/Infra |

**Golden rule for interviews:** 

- HPA and CA work together — HPA adds pods, and if there's no room to schedule them, CA adds nodes.

- VPA and HPA generally shouldn't be used on the same metric (CPU) together — they can conflict.

# Metrics Server (Prerequisite for HPA)

HPA needs a source of metrics. That source is the **Metrics Server**, not Prometheus (common confusion).

- Metrics Server collects resource usage (CPU/Memory) from the **kubelet's cAdvisor**.
- It exposes data via the **Metrics API** (`metrics.k8s.io`), NOT the core API.
- It is **not installed by default** in most clusters (kubeadm clusters need it added manually).

---

### Install Metrics Server (exam-relevant)
```bash
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
```
---

### Verify Metrics Server
```bash
kubectl top nodes
kubectl top pods
kubectl get apiservices | grep metrics
```

If `kubectl top` returns no data, check:
1. Metrics server pod is `Running` (`kubectl get pods -n kube-system`)
2. Logs for TLS/cert errors (`kubectl logs -n kube-system deploy/metrics-server`)
3. Pods actually have `resources.requests` set (HPA can't calculate % without requests)

# Horizontal Pod Autoscaler (HPA)

### Concept
HPA watches a metric (default: CPU utilization %) and changes the **replica count** of a Deployment/ReplicaSet/StatefulSet to keep the average metric near a target value.

**Formula HPA uses internally (know this cold for interviews):**
```
desiredReplicas = ceil( currentReplicas × ( currentMetricValue / desiredMetricValue ) )
```

Example: 3 replicas running at 90% CPU, target is 50%:
```
desiredReplicas = ceil(3 × (90/50)) = ceil(5.4) = 6
```

### Imperative creation (fast for exam)
```bash
kubectl autoscale deployment my-app --cpu-percent=50 --min=2 --max=10
```
---


### Key requirements for HPA to work (exam pitfalls)

- Target Deployment's pods **must have resources.requests.cpu** defined — HPA calculates % of the request, not an absolute number.
  
- metrics-server must be running and healthy.
  
- API version: autoscaling/v2 supports CPU + memory + custom/external metrics. autoscaling/v1 supports CPU only — CKA may still show v1 syntax in older docs, know both.

### Useful commands
```bash
kubectl get hpa
kubectl describe hpa my-app-hpa
kubectl get hpa my-app-hpa -o yaml
kubectl delete hpa my-app-hpa
```

`kubectl describe hpa` is the #1 debugging command — it shows current vs target metrics and recent scaling events.

### Behavior tuning (stabilization window) — good-to-know for interviews
Prevents flapping (rapid scale up/down):
```yaml
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 50
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15
```

# Vertical Pod Autoscaler (VPA)

### Concept
VPA adjusts a pod's **CPU/memory requests and limits** automatically based on historical usage, instead of scaling replica count. It is **not a built-in API** — it's a separate project you install via manifests (not part of core Kubernetes, so it's less emphasized in CKA but common in interviews).

### 3 Components of VPA
| Component | Job |
|---|---|
| **Recommender** | Watches usage, calculates recommended requests/limits |
| **Updater** | Evicts pods that need updated resources (so they get recreated with new values) |
| **Admission Controller** | Injects the new resource values into pods at creation time |

### Update Modes
```yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: my-app-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  updatePolicy:
    updateMode: "Auto"   # Auto | Recreate | Initial | Off
```

- **Off** → only gives recommendations, doesn't apply them (safe to run alongside monitoring).
- **Initial** → applies only at pod creation.
- **Recreate** → evicts and recreates pods with new values.
- **Auto** → currently behaves like Recreate (in-place resize is evolving in newer K8s versions).

### Why VPA + HPA (on CPU) don't mix
If VPA changes the CPU request and HPA is targeting CPU utilization %, they both react to each other's changes — causing instability. Use VPA on memory-bound workloads or with HPA on a **different metric** (e.g., custom/external metric) if you need both.

# Cluster Autoscaler (CA)

### Concept
CA runs as a pod in the cluster, watches for **pods that fail to schedule due to insufficient resources**, and calls the cloud provider's API to add/remove nodes.

- Scale-up trigger: a pod is `Pending` because no node has enough allocatable resources.
- Scale-down trigger: a node is underutilized for a set period **and** its pods can be safely moved elsewhere.

### Cloud-provider dependent
CA is **not cloud-agnostic** — it needs a cloud provider flag/plugin:
- AWS → integrates with Auto Scaling Groups (ASGs)
- GCP → integrates with Managed Instance Groups (MIGs)
- Azure → integrates with VM Scale Sets (VMSS)

This is why CA is rarely tested hands-on in CKA (it needs real cloud infra) but is **heavily asked in interviews**, especially for GCC/cloud-focused roles.

### Pods that block scale-down (know these for interviews)
- Pods with restrictive **PodDisruptionBudgets**
- Pods using **local storage** (emptyDir with data, hostPath)
- Pods not managed by a controller (bare pods)
- Pods with `cluster-autoscaler.kubernetes.io/safe-to-evict: "false"` annotation
- Kube-system pods (unless explicitly allowed)

### Key flags (good to mention in interviews)
```
--scale-down-unneeded-time=10m
--scale-down-utilization-threshold=0.5
--max-node-provision-time=15m
```

# HPA vs VPA vs CA — Comparison Table (Interview Favorite)

| Aspect | HPA | VPA | Cluster Autoscaler |
|---|---|---|---|
| What scales | Pod replica count | Pod resource requests/limits | Node count |
| Direction | Horizontal | Vertical | Infrastructure |
| Native to K8s core | Yes | No (separate project) | No (cloud-provider plugin) |
| Needs Metrics Server | Yes | Uses its own Recommender | No — reacts to scheduler events |
| Causes pod restart? | No (new pods added) | Yes (Recreate mode evicts pods) | No direct pod impact |

# Real-World Framing (from personal notes)

**Why HPA exists — plain-language version:**

- Some applications are business-critical and can't afford downtime or restarts (e.g., a checkout service).

- For these, **Horizontal Pod Autoscaling** adds more pod replicas automatically when load increases, and removes them identically when load drops — no restart of existing pods, no downtime.

**Why VPA exists — plain-language version:**

- Some applications *can* tolerate a restart/brief downtime. For these, instead of adding more pods, we **increase the resources (CPU/memory) of the existing pod** automatically. 

- This is useful when an app just needs to be "bigger," not "more."

**Cloud analogy (good interview soundbite):**

- In an on-prem/bare Kubernetes cluster, HPA/VPA scale **pods**. 

- In a cloud-managed setup, the same idea extends to **VMs/nodes** — when load increases, instead of just adding pods, the cluster can add more replicas *and* more nodes to keep cost optimum (no idle overprovisioned nodes sitting around).

---

**One-line distinctions to remember:**

- **HPA** → increased load → add more pod **replicas** → cost-optimal, uses only what's needed.
- **VPA** → increased load → **resize** the existing pod's resources → app must tolerate a restart.
- **Cluster Autoscaler** → increased load at the **infrastructure** level → node auto-provisioning, and this always comes bundled with/depends on a **cloud provider** (AWS/GCP/Azure) — you don't install this yourself the way you do VPA.
- **VPA is not built into Kubernetes by default** (unlike HPA, which is native) — it has to be installed separately as its own component.
- **HPA takes its metrics from the Metrics Server**, per pod, on a polling interval — this is the mechanical link between "load increases" and "HPA reacts."

### Event-driven autoscaling (KEDA) — production context

- Not all scaling decisions are CPU/memory-based. In real production systems, scaling is sometimes triggered by a **specific event**,
  
- e.g.:
 - Getting repeated `502` errors on the server
 - Queue length crossing a threshold
 - Custom application-level signals

This kind of **event-driven autoscaling** is commonly handled by a third-party tool called **KEDA (Kubernetes Event-Driven Autoscaling)**. 

KEDA extends HPA by feeding it custom/external metrics from event sources (message queues, HTTP error rates, cron schedules, etc.), rather than relying only on CPU/memory.

**Interview soundbite:** "Native HPA scales on CPU/memory by default. For event-driven or business-metric-based scaling — like scaling up because of a spike in 502 errors, or a queue backlog — we'd use KEDA, which plugs into HPA as a custom metrics source."

# CKA Exam Tips

1. **Practice creating HPA both imperatively and declaratively** — exam favors speed, use `kubectl autoscale` first, then verify/edit YAML if custom metrics needed.
2. Always check `kubectl top pods` works **before** creating HPA — if metrics-server is broken, HPA will show `<unknown>` for targets forever.
3. Remember: if pods don't have CPU **requests** set, HPA cannot compute utilization % — a very common trick question/scenario in mock exams.
4. `kubectl describe hpa <name>` is your best friend for troubleshooting during the exam.
5. VPA is **not part of the CKA curriculum objectives directly** — don't over-invest exam prep time here, but know it exists.
6. Cluster Autoscaler is **out of scope for hands-on CKA tasks** (it's infra/cloud-provider specific) — but understand it conceptually since CKA does test HPA + Metrics Server troubleshooting scenarios.
