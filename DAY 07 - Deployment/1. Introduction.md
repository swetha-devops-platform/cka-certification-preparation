# K8S Deployment

- A Deployment is a higher-level K8s object that manages pods for you.

- It creates a **ReplicaSet** under the hood, which constantly watches your pods and maintains the desired count

- **A Deployment manages the full lifecycle of pods — it creates a ReplicaSet that maintains the desired pod count, supports zero-downtime rolling updates by bringing new pods up before old ones go down, and allows instant rollback to a previous revision. You never use naked pods in production**

# Why We Moved to Deployments - This is the most important concept

- A Naked Pod can run an application, but Pods do not provide self-healing, scaling, rolling updates, or rollback capabilities.

- Deployments manage Pods through ReplicaSets and make applications production-ready. 

- That's why **Deployments** are used in production environments.

| Problem                  | What Happens with a Naked Pod?                                                               |
| ------------------------ | -------------------------------------------------------------------------------------------- |
| **Pod Crashes**          | The Pod is terminated and not automatically recreated.                                       |
| **Node Goes Down**       | All Pods running on that node are lost permanently.                                          |
| **No Scaling**           | Cannot maintain multiple replicas automatically (e.g., 3 copies of the application).         |
| **No Rolling Updates**   | Updating the container image requires deleting and recreating Pods, causing downtime.        |
| **No Rollback**          | If a faulty version is deployed, there is no built-in way to revert to the previous version. |
| **No Self-Healing**      | Kubernetes does not ensure the desired number of Pods are running.                           |
| **Not Production Ready** | Manual management becomes difficult as applications grow.                                    |


---

# What happens when a Pod crashes?

The ReplicaSet detects that the desired number of replicas is not met and automatically creates a new Pod to restore the desired state.

# Essential Deployment commands

```bash
# Create / apply
kubectl apply -f deployment.yaml

# Check status
kubectl get deployments
kubectl get pods
kubectl get replicaset            # see the auto-created ReplicaSet

# Describe — full detail + events
kubectl describe deployment my-app

# Scale up or down
kubectl scale deployment my-app --replicas=5

# Update image (triggers rolling update)
kubectl set image deployment/my-app my-app=nginx:1.22

# Watch rolling update in real time
kubectl rollout status deployment/my-app

# Rollback to previous version
kubectl rollout undo deployment/my-app

# See rollout history
kubectl rollout history deployment/my-app

# Delete
kubectl delete deployment my-app
```

# The 4 superpowers of Deployments

Now let's look at what makes Deployments essential — especially the rolling update, which is the most important


**1. Self-healing** — if a pod crashes, the ReplicaSet immediately creates a replacement. No manual intervention needed.

---

**2. Scaling** — one command changes the pod count instantly:

```bash
kubectl scale deployment my-app --replicas=10
```

---

**3. Rolling update (zero downtime)** — when you update the image, K8s brings new pods up one at a time before taking old ones down. Your app never goes offline. Controlled by two settings in your YAML:


```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1          # max extra pods above desired count during update
    maxUnavailable: 0    # max pods that can be unavailable during update
```
---

**4. Rollback** — bad deployment? One command reverts to the last working version:


```bash
kubectl rollout undo deployment/my-app
```
K8s keeps revision history automatically.
