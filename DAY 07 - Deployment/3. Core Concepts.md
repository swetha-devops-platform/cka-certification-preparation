## 1. The Problem: Pods Don't Self-Heal

On Kubernetes, we host our application as a container inside a Pod. But a bare Pod has no built-in resilience:

- If the container inside the Pod crashes, the application becomes unreachable
- Users get empty responses / connection failures
- There's no controller watching the Pod to bring it back

```
   ┌─────────────┐
   │    Node     │
   │  ┌───────┐  │        ✗
   │  │  Pod  │  │◄──────────► User
   │  │ nginx │  │   (connection fails
   │  └───────┘  │    when container crashes)
   └─────────────┘
```

**So how do we fix this?** We put a controller in front of the Pod whose entire job is to watch it and recreate it the moment it crashes.

---

## 2. ReplicationController (RC) — the original solution

- Responsible for making sure the desired number of replicas are running **at all times**
- The instant a Pod crashes, it spins up an identical replacement automatically
- Guarantees your application stays highly available

### Hands-on: `rc.yaml`

```yaml
apiVersion: v1
kind: ReplicationController
metadata:
  name: nginx-rc
  labels:
    env: demo
spec:
  replicas: 3
  template:
    metadata:
      name: nginx-pod
      labels:
        env: demo
    spec:
      containers:
        - name: nginx-container
          image: nginx
          ports:
            - containerPort: 80
```

**Why is there a `template` block?** 

Because the RC needs to know exactly what kind of Pod (which image, which labels, which ports) it's supposed to keep replicating. Without it, the controller wouldn't know what "identical replacement" means.

## Useful commands

```bash
kubectl apply -f rc.yaml          # create the RC (and its pods)
kubectl get rc                    # list replication controllers
kubectl get pods <podname>        # check pod status
kubectl describe pod <podname>    # full spec + events, used for troubleshooting
kubectl explain rc                # to learn about any field in the RC spec, on the fly
```

## 3. ReplicaSet (RS) — the recommended replacement for RC

| | ReplicationController | ReplicaSet |
|---|---|---|
| Status | Legacy | Recommended (current) |
| Selector | Equality-based only | Supports **set-based** selectors (`matchExpressions`) — more flexible |
| Managed by | N/A | Typically managed *by a Deployment*, not created directly |



**Key distinction to say out loud in an interview:** an RC/RS only manages the Pods it created as part of that RC/RS — it doesn't adopt arbitrary existing Pods unless their labels match its selector.


### `rs.yaml` — only real difference from RC is the `selector` block

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-rs
  labels:
    env: demo
spec:
  replicas: 3
  selector:
    matchLabels:
      env: demo
  template:
    metadata:
      name: nginx-pod
      labels:
        env: demo
    spec:
      containers:
        - name: nginx-container
          image: nginx
          ports:
            - containerPort: 80
```

**Selector, explained:**

`matchLabels: env: demo` means: *every Pod running with the label `env: demo` should be managed by this ReplicaSet.* This is how the RS finds and claims Pods — including ones that already exist and just happen to match.


**Note on apiVersion:** ReplicationController uses `apiVersion: v1`, but ReplicaSet (and Deployment) use `apiVersion: apps/v1` — a small detail that's an easy interview trip-up.


### Three ways to update/scale a ReplicaSet

1. **Edit the YAML file and re-apply**
   ```bash
   kubectl apply -f rs.yaml
   ```
2. **Edit the live object directly** (changes made in-cluster, not in your YAML file — so your file goes out of sync unless you copy the change back)
   ```bash
   kubectl edit rs/rs-name
   ```
3. **Imperative scaling** (no YAML editing at all)
   ```bash
   kubectl scale --replicas=10 rs/podname
   ```

## 4. Deployment — manages the ReplicaSet, for you

**Concept:** A Deployment manages ReplicaSets, and a ReplicaSet manages Pods. You never touch the ReplicaSet directly — the Deployment creates and controls it for you.



**Why do we need a Deployment at all, if RS already keeps Pods alive?**

- ReplicaSets keep your *current* Pods running — but they don't know how to safely *change* them. 

- Deployments add that extra functionality: versioned, zero-downtime updates.

**Example:** you have `nginx:1.1` running, and need to update to `nginx:1.2`.

- **If you used a bare ReplicaSet:** all Pods get recreated at once → users face downtime for that brief period
  
- **If you use a Deployment:** it performs a **rolling update** — it updates the first Pod, and while that Pod is updating, traffic is still served from the remaining old Pods. A new Pod also spins up to absorb the traffic that the updating Pod would have handled. At no point does capacity drop to zero.

### `deploy.yaml` — same shape as an RS, just change `kind`

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
  labels:
    env: demo
spec:
  replicas: 3
  selector:
    matchLabels:
      env: demo
  template:
    metadata:
      labels:
        env: demo
    spec:
      containers:
        - name: nginx-container
          image: nginx:1.1
          ports:
            - containerPort: 80
```

Kubernetes objects come with **sane defaults** — for example, a Deployment automatically gets a Service-compatible structure without you specifying every field by hand.


### Updating the image — by YAML or by command

**By YAML:** edit `image: nginx:1.1` → `image: nginx:1.2`, then `kubectl apply -f deploy.yaml`

**By command:**
```bash
kubectl set image deploy/deployname nginx=nginx:1.2
```

### Rollout history & rollback

```bash
kubectl rollout history deploy/deployname     # see all past revisions
kubectl rollout undo deploy/deployname         # roll back to the previous revision
```

### Other commands from today

```bash
kubectl get all      # see every k8s resource currently running in the namespace
```

---

**Interview-ready summary:** 

- A Pod alone has no self-healing — if it crashes, it's gone. A ReplicationController (legacy) or ReplicaSet (current, recommended) watches a set of Pods and recreates them automatically to maintain a desired replica count.

- A Deployment sits one level above a ReplicaSet — it manages ReplicaSets, and its job is to let you update your application (e.g. image version) with zero downtime using a rolling update strategy, while giving you rollback and rollout-history for free.
