# What are ConfigMaps and Secrets?

Applications require configuration to run, such as:

* Database URL
* API Endpoint
* Environment
* Username
* Password
* API Keys
* Tokens

Instead of hardcoding these values inside the application or container image, Kubernetes provides:

* **ConfigMap** → Store **non-sensitive configuration**
* **Secret** → Store **sensitive information**

This follows the **12-Factor App** principle of separating configuration from application code.

---

# Why do we need ConfigMaps and Secrets?

## Without ConfigMap

```text
Application
    │
Hardcoded Values
    │
Database URL
API URL
Environment
```

Problems:

* Need to rebuild the image for every configuration change.
* Same image cannot be reused across Dev, QA, and Production.
* Configuration is tightly coupled with the application.

---

## With ConfigMap

```text
Application
      │
      ▼
 ConfigMap
      │
Database URL
Environment
API Endpoint
```

Benefits:

* No code changes required.
* No image rebuild required.
* Same image can be deployed in multiple environments.

---

## Without Secret

```text
username = admin
password = admin123
apiKey = xxxxx
```

Problems:

* Credentials exposed.
* Security risk.
* Passwords stored in Git repositories.

---

## With Secret

```text
Application
      │
      ▼
 Kubernetes Secret
      │
Username
Password
API Keys
Tokens
```

Sensitive data is managed separately from the application.

---

# ConfigMap

## Definition

A **ConfigMap** is a Kubernetes object used to store **non-confidential configuration data** as key-value pairs.

Examples:

* Database Host
* Environment Name
* Log Level
* API URL
* Feature Flags

---

## ConfigMap Architecture

```text
Application
      │
      ▼
 Environment Variables
      │
      ▼
 ConfigMap
      │
Key-Value Pairs
```

---

# ConfigMap Example

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config

data:
  APP_ENV: production
  LOG_LEVEL: INFO
  DB_HOST: mysql-service
```

Create it:

```bash
kubectl apply -f configmap.yaml
```

View:

```bash
kubectl get configmaps

kubectl describe configmap app-config

kubectl get configmap app-config -o yaml
```

---

# Using ConfigMap as Environment Variables

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app

spec:
  containers:
  - name: nginx
    image: nginx

    envFrom:
    - configMapRef:
        name: app-config
```

All keys become environment variables inside the container.

---

# Using Individual Key

```yaml
env:
- name: DATABASE_HOST
  valueFrom:
    configMapKeyRef:
      name: app-config
      key: DB_HOST
```

---

# Mount ConfigMap as Volume

```yaml
volumes:
- name: config-volume
  configMap:
    name: app-config
```

Mount:

```yaml
volumeMounts:
- name: config-volume
  mountPath: /etc/config
```

Files created:

```text
/etc/config/APP_ENV
/etc/config/LOG_LEVEL
/etc/config/DB_HOST
```

Each key becomes a separate file.

---

# Secret

## Definition

A **Secret** is a Kubernetes object used to store **confidential information** such as passwords, tokens, certificates, and API keys.

Examples:

* Password
* Username
* OAuth Token
* SSL Certificate
* SSH Key

---

# Secret Architecture

```text
Application
      │
      ▼
 Environment Variables
      │
      ▼
 Kubernetes Secret
      │
Username
Password
Token
```

---

# Secret Example

Create Secret:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret

type: Opaque

data:
  username: YWRtaW4=
  password: YWRtaW4xMjM=
```

Notice:

```text
admin
```

becomes

```text
YWRtaW4=
```

This is **Base64 encoded**, **not encrypted**.

---

# Create Secret Imperatively

```bash
kubectl create secret generic db-secret \
--from-literal=username=admin \
--from-literal=password=admin123
```

---

# View Secret

```bash
kubectl get secrets

kubectl describe secret db-secret

kubectl get secret db-secret -o yaml
```

Decode:

```bash
echo YWRtaW4= | base64 --decode
```

Output:

```text
admin
```

---

# Using Secret as Environment Variable

```yaml
env:
- name: USERNAME
  valueFrom:
    secretKeyRef:
      name: db-secret
      key: username
```

---

# Using Secret as Volume

```yaml
volumes:
- name: secret-volume
  secret:
    secretName: db-secret
```

Mount:

```yaml
volumeMounts:
- name: secret-volume
  mountPath: /etc/secret
```

Files:

```text
/etc/secret/username

/etc/secret/password
```

---

# ConfigMap vs Secret

| Feature           | ConfigMap                         | Secret                                                         |
| ----------------- | --------------------------------- | -------------------------------------------------------------- |
| Purpose           | Store non-sensitive configuration | Store sensitive data                                           |
| Data Type         | Plain text                        | Base64 encoded                                                 |
| Examples          | URL, Port, Environment            | Password, API Key, Token                                       |
| Security          | No                                | Better than ConfigMap (supports encryption at rest if enabled) |
| Mounted as        | Env Variables / Volume            | Env Variables / Volume                                         |
| Kubernetes Object | Yes                               | Yes                                                            |

---

# Base64 Encoding

Many beginners think Secrets are encrypted.

They are **not**.

Example:

```text
admin
```

Encoded:

```text
YWRtaW4=
```

Anyone can decode:

```bash
echo YWRtaW4= | base64 --decode
```

Output:

```text
admin
```

For real security, enable **Encryption at Rest** and use external secret management solutions like **HashiCorp Vault**, **AWS Secrets Manager**, or **Azure Key Vault**.

# Useful Commands (CKA)

```bash
# ConfigMaps
kubectl get configmaps
kubectl describe configmap app-config
kubectl delete configmap app-config

# Secrets
kubectl get secrets
kubectl describe secret db-secret
kubectl delete secret db-secret

# Decode Secret
kubectl get secret db-secret -o jsonpath='{.data.password}' | base64 --decode
echo
```

# Troubleshooting (CKA)

### Pod cannot read ConfigMap

Check:

```bash
kubectl describe pod <pod-name>
```

Possible error:

```text
ConfigMap not found
```

Verify:

```bash
kubectl get configmaps
```

---

### Pod cannot read Secret

Check:

```bash
kubectl describe pod <pod-name>
```

Possible error:

```text
Secret not found
```

Verify:

```bash
kubectl get secrets
```

---

### Environment Variable Missing

Verify:

* ConfigMap/Secret name
* Key name
* YAML indentation
* Pod restart (if required)
---

# CKA Exam Tips

* Know how to create **ConfigMaps** and **Secrets** using both YAML and imperative commands.
* Understand how to consume them as:

  * Environment variables (`env`, `envFrom`)
  * Volumes
* Remember:

  * **ConfigMap → Non-sensitive data**
  * **Secret → Sensitive data**
* Be comfortable decoding Secret values using `base64`.
* Use `kubectl describe pod` to troubleshoot missing ConfigMaps or Secrets.

# Quick Revision Cheat Sheet

| Kubernetes Object | Stores                      | Common Examples                       | Can be Mounted As              |
| ----------------- | --------------------------- | ------------------------------------- | ------------------------------ |
| **ConfigMap**     | Non-sensitive configuration | DB Host, Port, Environment, Log Level | Environment Variables, Volumes |
| **Secret**        | Sensitive information       | Password, API Key, Token, Certificate | Environment Variables, Volumes |

---

**Memory Trick:**

- **ConfigMap = Configuration** → Non-sensitive settings.
- **Secret = Credentials**  → Passwords, tokens, certificates.
> Both are Kubernetes objects used to externalize application configuration and keep container images reusable.
