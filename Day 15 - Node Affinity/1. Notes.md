# Node Affinity 

**One-line summary:** 

Node Affinity is an advanced scheduling mechanism that uses node label expressions to require or prefer Pods to run on specific nodes — far more flexible than `nodeSelector`.

---

# What is Node Affinity?

- **Node Affinity** lets Pods be scheduled onto nodes based on **node labels**, using **rules and expressions** instead of a plain exact match.

- **Pod:** "I prefer or require running on nodes that match these label rules."

- Node Affinity is a **Pod property** that evaluates **node labels**.

---

#  Why Use Node Affinity?

- Schedule Pods on specific nodes based on labels
- Express complex/conditional scheduling rules
- Prefer certain nodes without making it mandatory
- Separate production, staging, and dev workloads
- Run GPU/SSD-dependent workloads on the right node type
- Overcome the limitations of `nodeSelector`

---

# Why Node Affinity Instead of Node Selector?

nodeSelector only supports **exact key-value matching**:

```yaml
nodeSelector:
  environment: production
```

It **cannot** express:
- Run on production **or** staging
- Do **not** run on development nodes
- Prefer SSD nodes but allow HDD if none available

Node Affinity solves all of these with operators and soft/hard rules.

---

# How Node Affinity Works

```text
Pod Created
      ↓
Scheduler reads affinity rules
      ↓
Checks node labels
      ↓
Evaluates expressions
      ↓
Finds matching nodes
      ↓
Schedules Pod
```

---

# Two Types of Node Affinity

| Type | Behaviour |
|---|---|
| requiredDuringSchedulingIgnoredDuringExecution | **Hard rule.** Must match or Pod stays Pending. |
| preferredDuringSchedulingIgnoredDuringExecution | **Soft rule.** Scheduler tries, but falls back if no match. |

---

## Required — Hard Rule

**Definition:** The Pod **must** be scheduled onto a matching node. If none match → **Pod remains Pending.**

**Example:**
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: environment
            operator: In
            values:
            - production
  containers:
  - name: nginx
    image: nginx
```

Result:
- Production node  - scheduled
- Development node -  not scheduled (stays Pending)

---

## Preferred — Soft Rule

**Definition:** Scheduler **tries** to place the Pod on matching nodes. If no matching node exists, the Pod is still scheduled elsewhere.

**Example:**
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  affinity:
    nodeAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 1
        preference:
          matchExpressions:
          - key: disk
            operator: In
            values:
            - ssd
  containers:
  - name: nginx
    image: nginx
```

Result:
- SSD node - preferred
- HDD node - still allowed if no SSD node is available

**Correction/addition — the weight field:** weight ranges from **1–100**. When multiple preferred terms are defined, the scheduler sums the weights of all terms a node satisfies, and favors nodes with the highest total score. This is a common exam/interview detail the original notes left unexplained.

---

# Understanding "IgnoredDuringExecution"

Very common interview question.

Suppose a Pod is scheduled on a node labeled environment=production. Later, the label is changed to environment=development. Will Kubernetes move or evict the Pod?

**No.**

IgnoredDuringExecution means the rule is checked **only at scheduling time**. Once the Pod is running, label changes on the node are ignored — the Pod stays put.

---

# Match Expressions (Operators)

Unlike nodeSelector, Node Affinity supports rich operators:

| Operator | Meaning | Example |
|---|---|---|
| **In** | Matches if the label value is in the list | environment In [production] matches environment=production |
| **NotIn** | Rejects the listed values | environment NotIn [development] → matches production/testing, rejects development |
| **Exists** | Checks only that the key exists, ignores value | gpu Exists matches gpu=true or gpu=false |
| **DoesNotExist** | Opposite of Exists — only nodes **without** the key match | — |
| **Gt** | Greater Than (numeric string comparison) | cpu Gt ["2"] matches cpu=4 |
| **Lt** | Less Than (numeric string comparison) | cpu Lt ["2"] matches cpu=1 |

**Correction:** `Gt`/`Lt` only work with matchExpressions (not matchFields), and the label value must be a valid integer string — they compare numerically, not lexicographically.

**Addition — matchExpressions vs matchFields :** most rules use matchExpressions against custom/label metadata. matchFields is a lesser-known variant that matches built-in node fields (e.g. metadata.name) rather than labels — rarely used but occasionally comes up as a "have you heard of this" interview question.

---

# Important Addition — AND/OR Logic (Common Exam Trap)

```yaml
nodeSelectorTerms:        # entries here are OR'd together
- matchExpressions:        # expressions inside ONE term are AND'd together
  - key: environment
    operator: In
    values: [production]
  - key: disk
    operator: In
    values: [ssd]
- matchExpressions:
  - key: environment
    operator: In
    values: [staging]
```

- Multiple nodeSelectorTerms → **logical OR** (node matches if it satisfies any term)
- Multiple matchExpressions inside the same term - **logical AND** (node must satisfy all of them)

---

# Node Selector vs Node Affinity

| Node Selector | Node Affinity |
|---|---|
| Simple | Advanced |
| Exact match only | Expression-based |
| No operators | `In`, `NotIn`, `Exists`, `DoesNotExist`, `Gt`, `Lt` |
| Mandatory only | Mandatory (required) + Soft (preferred) |

---

# Node Affinity vs Taints & Tolerations

| Node Affinity | Taints & Tolerations |
|---|---|
| Pod **chooses** nodes | Node **rejects** Pods |
| Uses labels | Uses taints |
| Placement control (attract) | Restriction control (repel) |
| Set by Pod owner | Set by Cluster Admin |

Both are commonly combined: **Affinity to attract** the right pods to a node, **taints to repel** the wrong ones — see the combined example below.

---

# Addition — Node Affinity vs Pod Affinity/Anti-Affinity

| Node Affinity | Pod Affinity / Anti-Affinity |
|---|---|
| Matches Pod against **node labels** | Matches Pod against **labels of other running Pods** |
| Controls which node a Pod lands on | Controls whether Pods are co-located or spread apart |
| Use case: "run on SSD nodes" | Use case: "run this pod near/away from another pod" (e.g. spread replicas across zones, or co-locate a cache with its app) |

---

# Node Affinity + Taints Together

Node labeled `gpu=true` **and** tainted `gpu=true:NoSchedule`.

```yaml
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: gpu
            operator: In
            values:
            - "true"
  tolerations:
  - key: gpu
    operator: Equal
    value: "true"
    effect: NoSchedule
```

- With both affinity match **and** matching toleration - Pod Scheduled
- Without the toleration - Pending (blocked by the taint even though affinity matched)

# CKA Exam Tips

- Node Affinity is an advanced version of`nodeSelector.
- Works with **node labels** via spec.affinity.nodeAffinity.
- requiredDuringSchedulingIgnoredDuringExecution → **hard rule**; no match = Pending.
- preferredDuringSchedulingIgnoredDuringExecution → **soft rule**; scheduler tries but can fall back.
- IgnoredDuringExecution → checked **only at scheduling time**; label changes post-scheduling don't evict.
- Operators: In, NotIn, Exists, DoesNotExist,Gt, Lt.
- Multiple nodeSelectorTerms = OR; multiple matchExpressions inside one term = AND.
- weight (1–100) only applies to preferred rules and determines priority among soft matches.
- Use kubectl describe pod to see the exact affinity-mismatch reason in Events.
