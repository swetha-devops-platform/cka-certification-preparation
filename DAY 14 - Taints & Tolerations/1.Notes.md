# Taints & Tolerations 

Taints repel Pods from Nodes, Tolerations allow Pods to tolerate those taints — but a toleration never *guarantees* scheduling.

# Core Concept

| | Taint | Toleration |
|---|---|---|
| Applied on | **Node** | **Pod** |
| Says | "Don't schedule any Pod on me unless it has permission" | "I can tolerate this node's restriction" |
| Controlled by | Cluster Admin | Pod Owner |
| Purpose | Repel/restrict | Permit (not force) |

**Node = taint. Pod = toleration.**

---

# Why Use Taints & Tolerations?

- Reserve nodes for specific workloads (e.g. GPU-only nodes)
- Isolate critical applications
- Separate production vs development workloads
- Prevent normal Pods from landing on special/dedicated nodes
- General scheduling control

---

# Real-World Example

Cluster with 4 nodes: `Node-1`, `Node-2`, `Node-3`, `GPU-Node`.

`GPU-Node` should only run ML workloads:

```bash
kubectl taint nodes GPU-Node gpu=true:NoSchedule
```

- Normal pods (`nginx`, `redis`, `mysql`) - **cannot** schedule on `GPU-Node`
  
- Only pods with a **toleration for `gpu=true`** - can run there

This matches the classic scenario: a pod trying to schedule on **Node A** (tainted `gpu=true`) is blocked unless it carries the matching toleration  - any pod without it simply lands on an untainted node like **Node B** instead.

---

# Taint Syntax

```
key=value:effect
```

Example:

```
environment=production:NoSchedule
```

| Part | Value |
|---|---|
| Key | `environment` |
| Value | `production` |
| Effect | `NoSchedule` |

---

# Taint Effects (3 total)

| Effect | New Pods (no toleration) | Existing Pods (no toleration) |
|---|---|---|
| **NoSchedule** | Blocked | Continue running |
| **PreferNoSchedule** | Avoided (soft) — may still run if no other node available | Continue running |
| **NoExecute** | Blocked | **Evicted** |

- **NoSchedule** → works on newer pods only; already-running pods are untouched.
- **PreferNoSchedule** → soft preference, no hard guarantee (scheduler avoids the node if possible).
- **NoExecute** → the strongest effect — blocks new pods *and* evicts existing ones without a matching toleration.

---

# How the Scheduler Uses a Taint

```
Node has taint (e.g. dedicated=monitoring:NoSchedule)
        ↓
   Pod arrives
        ↓
Does Pod tolerate the taint?
   YES → Schedule Pod
   NO  → Scheduling Failed (Pending)
```

---

# Toleration Syntax

```yaml
tolerations:
- key: "dedicated"
  operator: "Equal"
  value: "monitoring"
  effect: "NoSchedule"
```
---

# Toleration Operators

| Operator | Behaviour |
|---|---|
| **Equal** (default) | Matches both key **and** value exactly |
| **Exists** | Ignores value — only checks the key exists (matches `gpu=true` or `gpu=false`) |

---

# Does a Toleration *Force* Scheduling?

**No — very important interview point.**

- A toleration only **permits** a Pod to be scheduled onto a tainted node. It does **not force** placement. The scheduler still evaluates:

  - Available resources / resource requests & limits
  - Node affinity
  - Node selector
  - Other scheduling policies

- A toleration ≠ a guarantee. It only removes the taint as a blocker.

# Default Kubernetes Taints

Kubernetes auto-applies these to protect unhealthy nodes:

```
node.kubernetes.io/not-ready
node.kubernetes.io/unreachable
node.kubernetes.io/memory-pressure
node.kubernetes.io/disk-pressure
node.kubernetes.io/network-unavailable
```

# Hands-On Commands 

- **View taints on a node:**

```bash
kubectl get nodes
kubectl describe node <node-name>
kubectl describe nodes | grep Taints
```

- **Add a taint** (case-sensitive):
  
```bash
kubectl taint nodes <node-name> gpu=true:NoSchedule
```

- **Create a toleration on a pod** (generate YAML, then edit):

```bash
kubectl run redis --image=redis --dry-run=client -o yaml > redis.yaml
```
Under `spec`, add the toleration block:
```yaml
spec:
  tolerations:
  - key: "gpu"
    operator: "Equal"
    value: "true"
    effect: "NoSchedule"
```

- **Apply and verify:**
  
```bash
kubectl apply -f redis.yaml
kubectl get po -o wide
```

- **Remove a taint** (trailing `-` removes it):

```bash
kubectl taint nodes <node-name> gpu=true:NoSchedule-
```

- **Troubleshoot why a pod is stuck Pending:**

```bash
kubectl describe pod <pod-name>
```

  - check the **Events** section for untolerated-taint messages.

# CKA Exam Tips

- Taints → **Nodes**. Tolerations → **Pods**.
- A pod without a matching toleration **cannot** be scheduled on a NoSchedule node.
- NoExecute  both blocks new pods **and** evicts existing ones.
- PreferNoSchedule is a **soft** preference only.
- Tolerations never guarantee scheduling — they only make it possible.
- Control plane node is tainted NoSchedule by default.
- Use **kubectl describe node** for taints, **kubectl describe pod** for scheduling failures.
- Taint commands are **case-sensitive** — double-check key/value spelling in the exam.
