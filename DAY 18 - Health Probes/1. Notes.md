# Kubernetes Health Probes

# Why Probes Exist

Kubernetes doesn't know if your app is actually working — a pod can be "Running" while the app inside is deadlocked, still starting up, or stuck. Probes are how the kubelet checks app health and reacts.

---

# What are Health Probes?

---

---

- Health Probes are used by Kubelet to continuously monitor the health of containers.

- They help Kubernetes determine:

  - Is the application running?
  - Is the application ready to receive traffic?
  - Has the application started successfully?

- Without health probes, Kubernetes only knows whether the container process is running—not whether the application inside it is healthy.

---

# Why Health Probes are Important?

Imagine an application:

  - Container is running 
  - Java application inside is stuck 
  - Users cannot access the application 

- Without probes:

  - Kubernetes thinks everything is healthy.
  - Traffic continues to be sent to the broken Pod.

- With probes:

  - Kubernetes automatically detects the issue.
  - Restarts the Pod or removes it from Service endpoints.
 
---

# Types of Health Probes

| Probe           | Purpose                                              | Action Taken                         |
| --------------- | ---------------------------------------------------- | ------------------------------------ |
| Liveness Probe  | Checks whether application is alive                  | Restarts the container if failed     |
| Readiness Probe | Checks whether application is ready to serve traffic | Removes Pod from Service if failed   |
| Startup Probe   | Checks whether application has completed startup     | Prevents premature liveness failures |

**Interview one-liner:** Liveness restarts a broken container, readiness controls traffic routing, startup protects slow-starting apps from being killed prematurely by liveness.

---

# 1. Liveness Probe

## Purpose

Determines whether the application is still alive.

If the application becomes deadlocked or hangs:

* Kubelet restarts the container automatically.

---

## Real-world Example

Suppose:

* Spring Boot application hangs
* Container is still running
* Users receive timeout errors

Liveness Probe detects:

```
Application is unhealthy
```

Kubernetes performs:

```
Restart Container
```

---

## Flow

```
Container Running
       │
       ▼
Liveness Probe
       │
 Healthy?
   │        │
 Yes      No
 │          │
Keep      Restart
Running   Container
```

---

# YAML Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-liveness
spec:
  containers:
  - name: nginx
    image: nginx
    livenessProbe:
      httpGet:
        path: /
        port: 80
      initialDelaySeconds: 10
      periodSeconds: 5
```

---

# Important Parameters

## Key Fields

| Field                     | Explanation                                                                                                                                                                                                                                    | Default |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------: |
| **`initialDelaySeconds`** | **The amount of time Kubernetes waits after the container starts before performing the first health check.** This is useful when the application needs some time to initialize before it can respond to probes.                                          |     `0` |
| **`periodSeconds`**       | **The frequency at which Kubernetes performs the health check.** For example, if it's set to `10`, Kubernetes checks the application's health every 10 seconds.                                                                                          |    `10` |
| **`timeoutSeconds`**      | **The maximum time Kubernetes waits for a probe response before considering that probe as failed.** If the application doesn't respond within this time, the probe attempt is marked as a failure.                                                       |     `1` |
| **`successThreshold`**    | **The number of consecutive successful probe results required before Kubernetes considers the container healthy again.** This is mainly used with **Readiness** and **Startup** probes and must remain `1` for Liveness probes.                          |     `1` |
| **`failureThreshold`**    | **The number of consecutive failed probe attempts required before Kubernetes takes action.** For a **Liveness Probe**, it restarts the container. For a **Readiness Probe**, it marks the Pod as **NotReady** and removes it from the Service endpoints. |     `3` |

---

# What Happens if Liveness Fails?

```
Failure 1
Failure 2
Failure 3
        ↓
Container Restarted
```

The Pod remains the same, but the container inside it is restarted.

---

# 2. Readiness Probe

## Purpose

Determines whether the application is ready to receive requests.

If not ready:

* Pod stays running
* But Service stops sending traffic

No restart occurs.

---

## Real-world Example

Application is:

```
Loading cache...
Connecting to DB...
Loading configuration...
```

Application needs 30 seconds.

Without Readiness Probe:

Service sends traffic immediately.

Users receive:

```
500 Internal Server Error
```

With Readiness Probe:

```
Wait until application is ready
```

Then traffic begins.

---

## Flow

```
Pod Running
      │
      ▼
Readiness Probe
      │
 Ready?
 │         │
Yes       No
 │         │
Traffic   Remove from
Allowed   Service Endpoints
```

---

# YAML Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-readiness
spec:
  containers:
  - name: nginx
    image: nginx
    readinessProbe:
      httpGet:
        path: /
        port: 80
      initialDelaySeconds: 5
      periodSeconds: 5
```

---

# What Happens if Readiness Fails?

Container:

```
Still Running
```

Service:

```
Stops routing traffic
```

Pod:

```
Not Restarted
```

# 3. Startup Probe

## Purpose

Used for slow-starting applications.

Examples:

* Java
* Spring Boot
* Elasticsearch
* Kafka
* Jenkins

These applications may require several minutes to start.

---

## Problem Without Startup Probe

Suppose application needs:

```
2 minutes to start
```

Liveness Probe begins after:

```
10 seconds
```

Application isn't ready.

Liveness fails.

Container restarts repeatedly.

Result:

```
CrashLoopBackOff
```

---

## Startup Probe Solution

```
Startup Probe
      │
Application Started?
      │
 No
      │
Keep Waiting
      │
Yes
      │
Enable Liveness Probe
```

---

# YAML Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: startup-demo
spec:
  containers:
  - name: app
    image: nginx
    startupProbe:
      httpGet:
        path: /
        port: 80
      failureThreshold: 30
      periodSeconds: 10
```

This allows:

```
30 × 10 = 300 seconds

Application startup time
```

# Probe Methods

Kubernetes supports three probe types.

| Mechanism | How it works | Typical use |
|---|---|---|
| `httpGet` | GET request to a path/port; 200–399 = success | Web apps, REST APIs |
| `tcpSocket` | Attempts TCP connection to a port | Databases, non-HTTP services |
| `exec` | Runs a command inside the container; exit code 0 = success | Custom health check scripts |
| `grpc` | Calls gRPC health checking protocol (stable since v1.27) | gRPC services |


---

## HTTP Probe

Checks an HTTP endpoint.

```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
```

Common for:

* Spring Boot
* Node.js
* Python Flask
* Go APIs

---

## TCP Probe

Checks whether a TCP port is accepting connections.

```yaml
readinessProbe:
  tcpSocket:
    port: 3306
```

Common for:

* MySQL
* PostgreSQL
* Redis
* MongoDB

---

## Exec Probe

Runs a command inside the container.

```yaml
livenessProbe:
  exec:
    command:
    - cat
    - /tmp/healthy
```

Exit Code:

```
0 → Healthy

Non-zero → Failed
```

Useful when no HTTP endpoint exists.

```
        │

        │

Yes
        ▼
Readiness Probe
        │
Ready?
        │
Yes
        ▼

        │

        │
Yes

```

# Difference Between Liveness & Readiness

| Feature           | Liveness               | Readiness              |
| ----------------- | ---------------------- | ---------------------- |
| Purpose           | Checks if app is alive | Checks if app is ready |
| Failure Action    | Restart container      | Remove from Service    |
| Traffic Stopped   | No                     | Yes                    |
| Container Restart | Yes                    | No                     |
| Pod Deleted       | No                     | No                     |

---

# Difference Between Startup & Liveness

| Startup Probe                   | Liveness Probe                   |
| ------------------------------- | -------------------------------- |
| Used during application startup | Used after application starts    |
| Prevents premature restarts     | Detects runtime failures         |
| Runs once until success         | Runs continuously                |
| Best for slow-starting apps     | Best for monitoring healthy apps |

---

# Complete Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: health-probes-demo
spec:
  containers:
  - name: nginx
    image: nginx

    startupProbe:
      httpGet:
        path: /
        port: 80
      failureThreshold: 30
      periodSeconds: 10

    readinessProbe:
      httpGet:
        path: /
        port: 80
      initialDelaySeconds: 5
      periodSeconds: 5

    livenessProbe:
      httpGet:
        path: /
        port: 80
      initialDelaySeconds: 20
      periodSeconds: 10
```

# CKA Troubleshooting Tips

### Pod is Running but Service has no Endpoints

Check:

```bash
kubectl describe pod <pod-name>
```

Look for:

```
Readiness probe failed
```

---

### Pod Restarts Continuously

Check:

```bash
kubectl describe pod <pod-name>
```

Possible output:

```
Liveness probe failed
```

---

### Pod Stuck in CrashLoopBackOff

Possible reasons:

* Incorrect probe path
* Wrong port
* Application startup is slow
* Missing Startup Probe
* Probe timeout too short

---

### Events Help Identify Probe Failures

```bash
kubectl get events --sort-by=.metadata.creationTimestamp
```

Look for:

```
Readiness probe failed

Liveness probe failed

Back-off restarting failed container
```

---


# CKA Exam Tips

* Configure **liveness**, **readiness**, and **startup** probes using YAML.
* Remember the behavior:

  * **Liveness failure → Container restart**
  * **Readiness failure → Removed from Service endpoints**
  * **Startup failure → Startup retries until threshold; if exceeded, container is restarted**
* Know the three probe methods:

  * `httpGet`
  * `tcpSocket`
  * `exec`
* Use `kubectl describe pod` and `kubectl get events` to diagnose probe failures quickly during troubleshooting.

---

# Quick Revision Cheat Sheet

| Probe         | Purpose                        | Failure Result                               | Typical Use                 |
| ------------- | ------------------------------ | -------------------------------------------- | --------------------------- |
| **Liveness**  | Is the app still alive?        | Container restarted                          | Detect hung/deadlocked apps |
| **Readiness** | Can the app serve traffic?     | Removed from Service endpoints               | Delay or pause traffic      |
| **Startup**   | Has the app finished starting? | Container restarted after configured retries | Slow-starting applications  |

**Memory Trick:**

- **Startup → Start**  → Wait until the application finishes booting.
- **Readiness → Ready** → Control whether traffic reaches the Pod.
- **Liveness → Live**  → Restart the container if the application is unhealthy.
