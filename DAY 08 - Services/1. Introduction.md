# Kubernetes Services 

- A Kubernetes Service is an object that provides stable networking and load balancing for a set of Pods. 

- It enables communication between applications without depending on changing Pod IP addresses.

# Why Do We Need Services?

- Pods are ephemeral. Every time a pod restarts, dies, or gets rescheduled, it gets a brand new IP address. If your frontend is hardcoded to talk to 10.0.0.5 (your backend pod), and that pod restarts with IP 10.0.0.9, your entire app breaks. 

- This is the fundamental problem Services solve.

**Solution:** Instead of accessing application IP addresses directly, users access the **Service name**.

```
SVC ──────► IP Address
 ▲               │
 │               ▼
User Request   Service Name
```

- The Service forwards the request to the right pod. Kubernetes creates a DNS name automatically when you create a Service.

- **Example:** `payment-default.svc`

---

# How Services Work (Labels & Selectors)

- A Service doesn't track pod IPs. 

- Instead it uses label selectors — it continuously watches for any pod in the cluster that has the matching label. 

- This is why the **label in your Deployment YAML and your Service YAML must match exactly.**

- Since the label is the same for all pods, the Service works regardless of IP changes

```
                        ┌──────────────────────────┐
  ┌──────────────────┐  │  P1 ──┐                  │
  │ Deployment (yaml)│  │  P2 ──┤  same label      │
  │                  │──► RS   │  app: my-pay      │
  └──────────────────┘  │  P3 ──┘                  │
                        │  P4                      │
                        └──────────────────────────┘
```

In the YAML metadata, we assign a label:
```yaml
labels:
  app: my-pay
```

# Advantages of Services

1. **Load Balancing** — Distributes traffic across pods automatically

2. **Service Discovery** — Via Labels and Selectors (no need to track IPs)

3. **Expose to External World** — Can expose apps outside the cluster

# 3 Types of Kubernetes Services

---

**1. ClusterIP (Default)**

- Default service type

- Use this for internal communication between microservices.

- Advantages : Accessible **only inside** the Kubernetes cluster

- Disadvantages: Service Discovery, Load Balancing

```yaml
spec:
  type: ClusterIP
```

---

 **2. NodePort**

- Exposes the app **within your organisation** (inside network)

- Allows users inside the organisation or within the same network to access the application

- NodePort exposes on a port on each worker node's IP. It is accessible inside the organisation/network but **not securely** to the public internet.

```yaml
spec:
  type: NodePort
```

---

**3. LoadBalancer**

- Exposes the application to the **outside world** (public internet)

- **Works only on cloud platforms** (AWS, GCP, Azure)

- On **EKS (AWS):** Creates an **ELB (Elastic Load Balancer)** with a **public IP address**

- On **Minikube:** Use `minikube ip` to get the node IP

- LoadBalancer type only works on cloud providers. On local (Minikube), use NodePort for testing.

```yaml
spec:
  type: LoadBalancer
```

---

**To switch a service type:** - Change type: NodePort → type: LoadBalancer

```bash
kubectl edit svc <svc-name>
```

# Important Note

- **We should NEVER delete the default Kubernetes Service** (`kubernetes` in the `default` namespace). This is used by `kubeshark` and other tools to understand how traffic flows into containers within the cluster. - **kubectl get svc**

- **Why do we need multiple ReplicaSets?**  

Because there is always a user who accesses the application concurrently (at the same time). The number of ReplicaSets depends on:

  - The number of concurrent users

  - The number of connections a particular pod handles

- By default, pods don't have load balancing. You must **create a Service and add load balancing**. Pods are ephemeral — Services give them a stable identity.

# Service Architecture Summary

```
People within org  ──► NodePort   ──► Exposes app on Worker Node IP

People outside world ──► LoadBalancer ──► Public IP (cloud only)

Need public IP address → Use LoadBalancer
```

# Key Concepts Recap

| Concept | Description |
|---|---|
| **Service** | Acts as a load balancer and DNS entry point for pods |
| **Labels** | Key-value pairs assigned to pods in metadata |
| **Selectors** | Used by Services to find pods with matching labels |
| **ClusterIP** | Internal-only access (default) |
| **NodePort** | Access within org/network via node IP + port |
| **LoadBalancer** | Public internet access (cloud only) |
| **targetPort** | Port where the app runs inside the container |
| **kubeshark** | Tool for understanding traffic flow inside k8s |

# Common Commands

```bash
# View services
kubectl get svc

# Get detailed pod info (with API calls)
kubectl get pods -v=9

# Apply deployment or service
kubectl apply -f <filename>.yml

# Edit service (e.g., change type)
kubectl edit svc <svc-name>

# Minikube SSH and curl test
minikube ssh
curl -L http://localhost:<port>/<path> - For Golang Application 
```
