# Resource Requests & Limits

**One-line summary:** 

- Requests are what the Scheduler uses to placea Pod (minimum guaranteed resources); 

- Limits are what the Kubelet enforces at runtime (maximum allowed) — CPU over-limit gets throttled, Memory over-limit gets OOMKilled.

# What Are Resources?

Resources define **how much CPU and Memory a container needs, and how much it's allowed to use**.

Every Pod is really just an **encapsulation of one or more containers** — and inside each container, an actual process runs to do the real computing. That process is what needs CPU, memory, disk, and other resources to execute at all. Resources aren't an abstract Kubernetes construct; they map directly to what the process running inside the container physically consumes.

---

Two types:

| | Requests | Limits |
|---|---|---|
| Meaning | **Minimum** guaranteed resources | **Maximum** allowed resources |
| Used by | **Scheduler** (for Pod placement) | **Kubelet** (enforced at runtime) |

Resources are configured under the **container** section — **not** at the Pod level.

---

# Why Resources Matter

Without resource configuration:
- One Pod can consume all CPU and Memory on a node
  
- Other Pods slow down or fail
  
- The cluster becomes unstable
  
- The scheduler can't make informed placement decisions

---

**The core problem this solves:** if you don't specify requests/limits, you get insufficient resources errors and unpredictable behavior — that's exactly why Kubernetes asks you to declare them. Resources give you:

- Fair allocation across workloads
  
- A stable cluster
  
- Predictable performance

---

# Resource Requests

**Definition:** 

The **minimum** CPU/Memory guaranteed to a container. The **Scheduler** uses this — and only this — number to decide **how much is utilized, how much is still needed, and whether a node has room** before it places a Pod.

If no node has enough requested capacity → Pod stays **Pending**.

---

```yaml
resources:
  requests:
    cpu: "500m"      # guaranteed 0.5 CPU core
    memory: "256Mi"  # guaranteed 256 MB RAM
```

---

# Resource Limits

**Definition:** 

The **maximum** CPU/Memory a container is allowed to consume, enforced by the **Kubelet** at runtime (the scheduler doesn't check this — only requests).

---

| If usage exceeds the limit | CPU | Memory |
|---|---|---|
| Result | **Throttled** — container keeps running, just slower | **OOMKilled** — container is terminated (and restarted per policy) |
| Why the difference | CPU is *compressible* — can be rationed without stopping the process | Memory is *incompressible* — can't be throttled, only reclaimed by killing the container |

---

```yaml
resources:
  limits:
    cpu: "1"        # cannot exceed 1 CPU core
    memory: "512Mi" # cannot exceed 512 MB RAM
```

---

# The Scenario Without Requests/Limits (Why This Matters in Practice)

- If a container needs CPU and memory but **you never specify requests or limits** on it:

  - You haven't told Kubernetes how much it should take, or where the boundary is
  
  - The container isn't bound to any resource boundary at all
  
  - As soon as load increases, it can try to grab the **entire node's** CPU and memory
  
  - Result: the Pod crashes with an **OOM error** — literally running out of memory because nothing was capping it

- This is the real-world reason requests/limits exist: 

- **without them, a single misbehaving Pod can starve or crash others on the same node** — not just itself. 

- Setting requests/limits is what keeps that blast radius contained to the Pod, rather than the whole node.

---

# Complete Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx
    resources:
      requests:
        cpu: "250m"
        memory: "128Mi"
      limits:
        cpu: "500m"
        memory: "256Mi"
```

---

# Scheduler Behavior

The Scheduler checks **CPU & Memory Requests only** — never Limits.

Example:
```
Node Capacity:   CPU = 2 Core, Memory = 4Gi
Pod Request:     CPU = 500m,  Memory = 1Gi
```
Scheduler verifies the node has **at least** 500m CPU and 1Gi memory free → if yes, schedules the Pod.

---

# Runtime Behavior (Recap)

**CPU exceeds limit:**
```
Application slows down.
CPU is throttled.
Pod is NOT killed.
```

**Memory exceeds limit:**
```
OOMKilled
Container restarts (per restartPolicy)
```
Memory cannot be throttled — only killed and restarted.

# CPU Units

```
1     = 1 full CPU core
1000m = 1000 milliCPU = 1 core
```

| Value | Meaning |
|---|---|
| 100m | 0.1 CPU |
| 250m | 0.25 CPU |
| 500m | 0.5 CPU |
| 1000m | 1 CPU |
| 2000m | 2 CPU |

---

# Memory Units

Common units: `Ki`, `Mi`, `Gi`

Examples: `128Mi`, `256Mi`, `512Mi`, `1Gi`, `2Gi`

---

# If Requests Are Missing

- Scheduler assumes **0** unless a **LimitRange** sets a namespace default
- Pod can get scheduled onto an already-busy node unexpectedly
- No resource guarantee is provided to the container

# If Limits Are Missing

- Container can consume all available CPU/Memory on the node (subject to any cluster-level policies like `LimitRange`/`ResourceQuota`)
- Directly enables the "one Pod eats the whole node" scenario from Section 5

---

# Requests vs Limits — Quick Reference

| Requests | Limits |
|---|---|
| Minimum guaranteed resources | Maximum allowed resources |
| Used by **Scheduler** | Enforced by **Kubelet** |
| Determines Pod placement | Controls runtime usage |
| Pod stays Pending if unavailable | CPU throttled / Memory OOMKilled if exceeded |

# CKA Hands-On Commands

**Check configured requests/limits on a running pod:**
```bash
kubectl describe pod <pod-name>
```
Shows the `Requests:` and `Limits:` block per container.

**Check node capacity:**
```bash
kubectl describe node <node-name>
```
Shows `Capacity`, `Allocatable`, and `Allocated Resources`.

**Live usage (requires Metrics Server):**
```bash
kubectl top pod
kubectl top node
```

**Edit resources on a running workload:**
```bash
kubectl edit deployment nginx
```
or update the YAML and:
```bash
kubectl apply -f deployment.yaml
```

**Generate a Pod YAML imperatively, then add resources:**
```bash
kubectl run nginx --image=nginx --dry-run=client -o yaml > pod.yaml
```
Add under the container spec:
```yaml
resources:
  requests:
    cpu: "250m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "256Mi"
```
```bash
kubectl apply -f pod.yaml
```

---

# CKA Exam Tips

- Resources are defined **per container**, not at the Pod level.
- Scheduler uses **Requests only** — never checks Limits.
- CPU over limit → **throttled**, container keeps running.
- Memory over limit → **OOMKilled**, container terminated.
- No requests set → scheduler treats it as needing 0 (unless a LimitRange default applies) — can lead to unpredictable placement.
- No limits set → container can consume the whole node, endangering other Pods.
- kubectl describe pod shows configured requests/limits; kubectl top pod/kubectl top node show live usage and need Metrics Server installed.
