# Kubernetes DaemonSets

# 30-Second Interview Answer

A **DaemonSet** ensures exactly **one Pod runs on every eligible node** in the cluster — no `replicas` field needed. When a node joins, the DaemonSet controller schedules a Pod there automatically; when a node leaves, that Pod is deleted with it. It's the go-to workload for node-level agents: logging, monitoring, networking (CNI), storage (CSI), and security agents.

---

# What Is a DaemonSet?

A DaemonSet is a controller that guarantees **one Pod per eligible node** — not a fixed replica count like a Deployment.

- **New node joins** → DaemonSet controller detects it → schedules a Pod automatically.
- **Node removed** → its DaemonSet Pod is deleted automatically.
- **No manual scaling** — `kubectl scale` doesn't work on a DaemonSet. Pod count = eligible node count, always.

This is the core difference from a Deployment: a Deployment lets *you* decide the replica count (and it can pile multiple replicas onto one node); a DaemonSet lets the **cluster topology decide**, capped at **one Pod per node**.

---

# Why Do We Need DaemonSets? (Use Cases)

Some workloads have to run on *every* node, not just a few:

| Category | Purpose | Examples |
|---|---|---|
| **Logging** | Every node produces its own logs; a collector must run locally to ship them | Fluent Bit, Fluentd, Filebeat |
| **Monitoring** | Pulls CPU/memory/disk/network metrics per node | Node Exporter, Datadog Agent, New Relic Agent |
| **Networking (CNI)** | Pods can't talk to each other without this running on every node | Flannel, Calico, Cilium, Weave Net |
| **Storage (CSI)** | Mounts volumes locally on each node | EBS CSI Driver, Azure Disk CSI, GCE PD CSI |
| **Security** | Continuous runtime scanning per node | Falco, CrowdStrike, Aqua Security |

**Interview framing:** kube-proxy itself is a great real-world example — it's deployed as a DaemonSet so that every node can handle its own Service routing/iptables rules locally.

---

# DaemonSet vs Deployment

| Deployment | DaemonSet |
|---|---|
| Runs a specified number of replicas | Runs one Pod per eligible node |
| You control replica count | Kubernetes controls Pod count based on nodes |
| Used for stateless applications | Used for node-level system services |
| Multiple Pods can land on one node | Max **one** Pod per node |
| Scale manually (`kubectl scale`) | Scales automatically as nodes are added/removed |

---

# Update Strategies

| Strategy | Behavior |
|---|---|
| **RollingUpdate** (default) | Updates Pods node-by-node automatically, no downtime |
| **OnDelete** | Pod only updates after you manually delete it — full admin control |

```yaml
updateStrategy:
  type: RollingUpdate
```

---

## DaemonSet - Useful `kubectl` Commands

| **Command**                                 | **Description**                                                                                                                                           |
| ------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `kubectl apply -f daemonset.yaml`           | Create or update a DaemonSet from a YAML manifest.                                                                                                        |
| `kubectl get ds`                            | List all DaemonSets in the current namespace.                                                                                                             |
| `kubectl get daemonsets`                    | Full command to list DaemonSets in the current namespace.                                                                                                 |
| `kubectl get ds -A`                         | List DaemonSets across **all namespaces**. Useful because system DaemonSets (e.g., `kube-proxy`, CNI plugins) usually run in the `kube-system` namespace. |
| `kubectl get ds -n kube-system`             | List DaemonSets only in the `kube-system` namespace.                                                                                                      |
| `kubectl describe ds fluent-bit`            | Display detailed information about the `fluent-bit` DaemonSet, including events, selectors, update strategy, and Pod status.                              |
| `kubectl get ds fluent-bit -o yaml`         | View the complete YAML definition of the `fluent-bit` DaemonSet.                                                                                          |
| `kubectl get pods -o wide`                  | Display Pods with additional details such as the node each Pod is running on. Useful for verifying that one DaemonSet Pod exists on each eligible node.   |
| `kubectl delete ds fluent-bit`              | Delete the `fluent-bit` DaemonSet and all Pods managed by it.                                                                                             |
| `kubectl config get-contexts`               | List all available Kubernetes contexts configured on your machine.                                                                                        |
| `kubectl config use-context <context-name>` | Switch to a different Kubernetes cluster/context. Useful when practicing on multiple clusters.                                                            |

 New Worker-3 joins → controller detects it → Daemon Pod created automatically

# CKA Exam Tips

- DaemonSet creates **one Pod per eligible node** — never a `replicas` field.
- New nodes automatically get a Pod; removed nodes automatically lose theirs.
- Control-plane nodes are tainted — a DaemonSet Pod needs a **toleration** to schedule there.
- Always check `-A` or the right `-n` namespace; don't assume `default`.
- Common use cases: logging, monitoring, networking (CNI), storage (CSI), security.
- Default update strategy is **RollingUpdate**; `OnDelete` is the manual-control alternative.
- `kubectl scale` does **not** work on DaemonSets.

---

# Closing Line

Think of a Deployment as **"run N copies, I'll decide how many."** A DaemonSet flips that logic: **"run exactly one copy everywhere I'm needed — you decide by counting nodes, not me."** That's the whole mental model in one sentence.
