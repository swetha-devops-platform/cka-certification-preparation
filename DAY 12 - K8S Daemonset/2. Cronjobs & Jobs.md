# Kubernetes Jobs & CronJobs 

**30-Second Interview Answer**

A **Job** runs a Pod to completion exactly once — it's for a task that finishes and exits, not a long-running service. A **CronJob** is a Job that's triggered on a **time-based schedule** using standard cron syntax. Both are "run to completion" workloads, unlike Deployments/DaemonSets, which are meant to keep running forever.

---

# Job -  What is it?

A **Job** creates one or more Pods and makes sure they run **until successful completion** — then it stops. It doesn't keep the Pod alive afterward like a Deployment would.

- Runs **once**, finishes, done.
- If the Pod fails, the Job retries it (based on `backoffLimit`).
- No `replicas`, no continuous reconciliation loop to "keep N running" — it's about **completion**, not uptime.

---

# When do you use it?

- One-time **installation or provisioning scripts** (e.g., setting up a new cluster component)
- **Database migrations**
- **Batch processing** — process a file, then exit
- Anything you'd otherwise run manually once and forget

---

# CronJob - What is it?

A **CronJob** wraps a Job with a **schedule** — it creates a new Job automatically at a defined time, on a recurring basis. Think of it as "Job + cron timer."

---

# Why do we need it? (Use Cases)

- Generating **reports** at a fixed time — daily, weekly, monthly
- **Cleanup jobs** — clearing old logs, temp files, expired data
- Any recurring, time-based automation you'd otherwise set up as a cron on a VM

---

# Cron Schedule Syntax

```
*  *  *  *  *
│  │  │  │  └── Day of week (0-6, Sunday = 0)
│  │  │  └───── Month (1-12)
│  │  └──────── Day of month (1-31)
│  └─────────── Hour (0-23)
└────────────── Minute (0-59)
```

Set with the `schedule:` field, e.g. `schedule: "*/20 * * * *"`.

**Interval shorthand — the interview-friendly trick:** `*/n` means "every n units of that field."
- Every 20 minutes → `*/20 * * * *`
- Every 6 hours → `0 */6 * * *`
- Every day at 2 AM → `0 2 * * *`
- Every Sunday at midnight → `0 0 * * 0`

---

# Job vs CronJob

| Job | CronJob |
|---|---|
| Runs once, to completion | Runs repeatedly, on a schedule |
| Triggered manually / by `kubectl apply` | Triggered automatically by the cron schedule |
| No `schedule` field | Requires `schedule` field (cron syntax) |
| Used for one-off tasks (install, migrate, provision) | Used for recurring tasks (reports, cleanup) |

## kubectl Commands Cheat Sheet

---

| **Command**                                                     | **Description**                                                                                                                          |
| --------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `kubectl apply -f job.yaml`                                     | Create or update a Job from the YAML manifest.                                                                                           |
| `kubectl get jobs`                                              | List all Jobs in the current namespace.                                                                                                  |
| `kubectl describe job db-migration`                             | Display detailed information about the `db-migration` Job, including status, events, completions, and Pod details.                       |
| `kubectl logs job/db-migration`                                 | View the logs of the Pod created by the `db-migration` Job.                                                                              |
| `kubectl delete job db-migration`                               | Delete the `db-migration` Job and its associated Pods (unless configured otherwise).                                                     |
| `kubectl apply -f cronjob.yaml`                                 | Create or update a CronJob from the YAML manifest.                                                                                       |
| `kubectl get cronjobs`                                          | List all CronJobs in the current namespace.                                                                                              |
| `kubectl get cj`                                                | Short form of `kubectl get cronjobs`.                                                                                                    |
| `kubectl describe cronjob log-cleanup`                          | Display detailed information about the `log-cleanup` CronJob, including its schedule, history, and recent executions.                    |
| `kubectl get jobs --watch`                                      | Continuously monitor Jobs as they are created and updated. Useful for watching Jobs spawned by a CronJob.                                |
| `kubectl create job --from=cronjob/log-cleanup manual-test-run` | Manually create a Job from the `log-cleanup` CronJob. Useful for testing the CronJob immediately without waiting for the scheduled time. |

# CKA Exam Tips

- `restartPolicy` on a Job/CronJob pod must be `Never` or `OnFailure` — **not** `Always`.
- `*/n` in any cron field means "every n units" — this is the fastest way to write intervals under exam time pressure.
- A CronJob's pod spec lives under `spec.jobTemplate.spec.template.spec` — one extra nesting level versus a plain Job.
- Use `kubectl create job --from=cronjob/<name>` to manually fire a CronJob's schedule for testing, without waiting for the actual time to hit.
- `backoffLimit` controls retries on failure — know this field, it's a common exam trip-up.

---


# Closing Line

If a Deployment is a service that never stops, and a DaemonSet is a service on every node, then a **Job is a task that finishes**, and a **CronJob is that same task on an alarm clock**. Once you see "run once" vs "run forever" vs "run forever on every node," the whole workload-type family falls into place.
