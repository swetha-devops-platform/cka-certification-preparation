# Static Pods

## 30-Second Interview Answer

- Normally the **scheduler** decides which node a Pod runs on. But the scheduler itself is a control-plane component running as a Pod — so who schedules it? 
- Answer: nobody. It's a **Static Pod**, created directly by the **kubelet** from a local manifest, completely bypassing the scheduler.
---


# The Problem They Solve

- The **scheduler** is the control-plane component responsible for deciding which node a Pod lands on, based on its scheduling algorithm
  
- it watches for unscheduled Pods and hands the node assignment to the API server, which tells the kubelet to run it.

- But here's the paradox: **the scheduler itself runs as a Pod.**

- If Pods need the scheduler to be scheduled, what schedules the scheduler? This is exactly why **Static Pods** exist — they're control-plane components that are **not managed or scheduled by the scheduler at all**.

---

# What is a Static Pod?

A Static Pod is managed **directly by the kubelet**, not by the API server or scheduler.

- The kubelet continuously watches a directory on the node (`staticPodPath`) for Pod manifest files.
- Add a manifest → kubelet creates the Pod. Modify it → kubelet updates it. Remove it → kubelet deletes it.
- A **Mirror Pod** is reflected into the API server purely for visibility — you can `kubectl get`/`describe` it, but you can't manage it through the API server.
- Static Pods are commonly used to run Kubernetes control plane components

**Used for exactly the control-plane components that can't depend on the scheduler to exist first:**
- kube-apiserver
- kube-scheduler
- kube-controller-manager
- etcd
  
---

# Static Pod Manifest Location

- **Default directory kubelet watches** - cd /etc/kubernetes/manifests

- **Confirm it from the kubelet's own config** -  ps -ef | grep kubelet & cat /var/lib/kubelet/config.yaml

- **look for: staticPodPath** - cd /etc/kubernetes/manifests


This directory is exactly where you'd go to troubleshoot control-plane components — editing one of these YAMLs and saving it is how you'd fix a broken `kube-apiserver` or `etcd` on the exam.

# Create / Delete a Static Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-static
spec:
  containers:
  - name: nginx
    image: nginx
```

Save it as `/etc/kubernetes/manifests/nginx-static.yaml` — within seconds, `kubectl get pods` shows it running.

```bash
kubectl delete pod nginx-static     # kubelet recreates it instantly — the manifest still exists
rm /etc/kubernetes/manifests/nginx-static.yaml   # this is the only way to actually remove it
```

**Hands-on insight:** if you delete the manifest for something like the scheduler itself and then try to create a new Pod, that new Pod won't run — but any Pod that was already running keeps running until it eventually fails/restarts. The manifest file is the single source of truth kubelet keeps re-checking against.

# Getting a Shell on the Node (Kind / Minikube setups)

If you're on a real cloud VM cluster, you'd just `ssh` or `rdp` into the node. On a **Kind or Minikube** setup, each "node" is actually a Docker container, so you get in with `docker exec` instead:

```bash
docker ps                          # find the node's container name
docker exec -it <NodeName> bash    # or sh, if bash isn't available
```

This is how you'd go poke around `/etc/kubernetes/manifests` on a local practice cluster.

### Static Pod vs Normal Pod

| Feature | Static Pod | Normal Pod |
|---|---|---|
| Managed by | Kubelet | API Server |
| Created using | Local manifest file | `kubectl apply` |
| Scheduler involved | No | Yes |
| Runs on | Only the node with the manifest | Any node the scheduler picks |
| Auto-restart | Yes | Yes |
| Manageable by Deployment/ReplicaSet | No | Yes |
