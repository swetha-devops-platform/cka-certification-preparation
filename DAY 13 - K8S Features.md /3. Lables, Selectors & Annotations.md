# Labels, Selectors & Annotations

# Labels

**Key-value pairs** attached to an object's `metadata`, used to identify, group, and filter resources.

```yaml
metadata:
  labels:
    app: nginx
    env: production
    tier: frontend
```

- Labels are how Kubernetes objects find each other: a Service's selector matches a Pod's labels; a ReplicaSet's matchLabels matches the Pods it should manage.

**Important nesting detail:** 

- in a Deployment/ReplicaSet manifest, the label under top-level `metadata` describes *the Deployment object itself*.

- The label under `spec.template.metadata` is what's actually stamped onto the **Pods**, and that's what `spec.selector.matchLabels` has to match not the Deployment's own top-level label.

--- 

# Selectors

Selectors are **queries against labels** — how a resource says "act on anything with this label."

```yaml
# Pod
metadata:
  labels:
    app: nginx

# Service
spec:
  selector:
    app: nginx        # routes traffic only to Pods labeled app=nginx
```
---

**Types:**

- **Equality-based:** `app=nginx`, `env!=prod`
  
- **Set-based:** `env in (prod,dev)`, `env notin (test)`, `tier exists`

---

# Annotations (the other kind of metadata)

Labels are for **identifying and querying** — annotations are for **storing extra descriptive information** that isn't meant to be queried/filtered on (build version, contact info, changelog notes, tooling metadata, etc.).

| | Labels | Annotations |
|---|---|---|
| Purpose | Identify & select objects | Attach non-identifying extra info |
| Queryable via `-l` / selectors | Yes | No |
| Used by Services/Deployments to match Pods | Yes | No |

---

# Where Labels Show Up

Pods, Deployments, ReplicaSets, Services, StatefulSets, DaemonSets, Jobs, CronJobs, and Nodes — basically every object type.

---

# kubectl Commands Cheat Sheet

```bash
kubectl get pods --show-labels                     # see all labels on every pod
kubectl get pods -l app=nginx                       # filter by one label
kubectl get pods -l app=nginx,env=prod              # filter by multiple labels
kubectl get pods --selector tier=frontend           # same idea, --selector form
kubectl label pod nginx env=dev                     # add a label
kubectl label pod nginx env-                        # remove a label
```

**Hands-on flow to practice:** create two `pod.yaml`s with different labels/selectors, then use `kubectl get pod --selector tier=frontend` to pull back only the frontend-tagged one. This is exactly how a Service picks its backend Pods under the hood.

# CKA Exam Tips — All Three Topics

- **Static Pods:** know `/etc/kubernetes/manifests`, that kubelet — not the API server — owns them, and that deleting the manifest (not just the Pod) is the only real way to remove one.
  
- **Manual Scheduling:** practice setting `nodeName` directly in a Pod spec; know that `nodeName` always overrides `nodeSelector` since the scheduler is bypassed entirely.
  
- **Labels/Selectors:** be fast with `-l`/`--selector` filtering and comfortable writing `matchLabels` under a ReplicaSet/Deployment's `spec.selector` — this is one of the most frequently tested fundamentals in both the CKA exam and DevOps interviews.

---

## Closing Line

All three of these topics answer the same underlying question — **"who's actually in control here?"** Static Pods: the kubelet, not the scheduler. Manual scheduling: you, not the scheduler. Labels & selectors: whichever resource is querying, not some central registry. Once you see that thread, the "why" behind each concept sticks a lot better than the syntax alone.
