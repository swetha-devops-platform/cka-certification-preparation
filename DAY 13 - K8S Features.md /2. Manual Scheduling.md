# Manual Scheduling

## What Is It?

- Normally: Pod created → API Server → Scheduler picks a node → Kubelet on that node runs it.

- **Manual scheduling skips the middle step** — you tell Kubernetes exactly which node to use yourself, via nodeName.

---

# Method 1: nodeName (the common way)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  nodeName: worker01
  containers:
  - name: nginx
    image: nginx
```

The kubelet on `worker01` picks this up and runs it immediately — the scheduler never even sees this Pod as unscheduled, because it already has a node assigned.

**Hands-on / practical flow:**

```bash
kubectl run nginx --image=nginx -o yaml --dry-run=client > pod.yaml
# open pod.yaml, add nodeName under spec (sibling of containers)

kubectl get nodes                  # copy the exact node name from here
# paste it in as the nodeName value
```
---

# Method 2: Scheduler disabled entirely (exam scenario)

- If the scheduler is down and you create a Pod **without** nodeName, it sits in Pending state forever — no one is watching for it.

- You bind it manually:

```bash
kubectl replace --raw "/api/v1/namespaces/default/pods/nginx/binding" -f binding.yaml
```

```yaml
apiVersion: v1
kind: Binding
metadata:
  name: nginx
target:
  apiVersion: v1
  kind: Node
  name: worker01
```

---

**The key mental model from hands-on practice:** with two Pods side by side — one with `nodeName` set, one without — the scheduler only "cares about" and picks up the one *without* a `nodeName`. The one with `nodeName` already set bypasses it entirely and goes straight to `Pending → Running` via the kubelet on that node.

---

# Manual Scheduling vs Normal Scheduling

| Feature | Scheduler | Manual Scheduling |
|---|---|---|
| Chooses the node | Yes | No — you do |
| Uses scheduling policies/algorithm | Yes | No |
| Requires `nodeName` | No | Yes |
| Common in production | Yes | Rare — mostly exam/troubleshooting |
